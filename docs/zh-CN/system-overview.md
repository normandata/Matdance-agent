# 系统在做什么

Language: [English](../system-overview.md) | 中文

Matdance 的核心不是“让模型说得更像人”，而是给 agent 一个能长期工作的本地操作系统。这个操作系统不修改模型权重，也不假装自己拥有云端平台的复杂权限模型；它做的是把一次次会话产生的状态、经验、文件、工具结果和后台维护记录固化下来，让下一次协作有东西可查、有东西可修、有东西可迁移。

## 会话变成状态

普通聊天系统最脆弱的地方，是它把一切都押在当前上下文窗口里。窗口足够长时看起来无事发生；窗口一压缩，很多细节就像从没存在过。Matdance 的做法是把会话拆成几类状态：

- `sessions/*.json` 保存会话统计、上下文使用量、活动时间和任务状态。
- `sessions/*.state.json` 保存消息、工具调用、thinking/reasoning、提示音卡片、文件预览 marker 和当前 active task。
- `workspace/` 保存 agent 产出的文件，让聊天不是唯一的交付载体。
- `runtime/events/`、`scheduled_tasks/runs/` 保存后台任务和恢复链路的证据。

因此，Matdance 的聊天不是一个只向上游模型转发消息的窗口。每次工具调用、每次任务推进、每次后台整理，都会成为之后可以检查的本地事实。

## 记忆变成分层资产

Matdance 不把“记忆”设计成一个无限增长的 prompt。它按稳定程度分层：

- Hot memory 记录近期状态和当前协作上下文，会注入 prompt，类似桌面上的工作便签。
- Core memory 记录稳定偏好、长期规则、项目约定和反复会用到的经验，会注入 prompt，类似常备手册。
- Long-term memory 按日期保存详细档案，不默认注入 prompt，类似图书馆里的文献。
- Vector memory 是本地检索索引，用来从这些 Markdown 资产里找线索，不取代原文。

这套结构和上下文压缩互补。压缩负责让当前会话继续跑，记忆整理负责把值得保留的事实迁移到更合适的位置。它不能保证每个字都被保存，但在正常运行条件下，会显著降低“会话里发生过但之后完全无从追溯”的概率。

长期使用时，不建议把所有工作都塞进一个永远不结束的会话。阶段性任务完成后，或者大约一周左右，开新会话继续协作通常更稳。跨会话记忆会保留重要信息，新会话则给整理系统一个更干净的增量边界，减少旧会话过长带来的整理压力。

## 技能变成可验证流程

技能系统保存的不是百科知识，而是未来 agent 可以复用的操作方法。一个合格技能应该回答这些问题：什么时候使用、需要什么前提、具体步骤是什么、依赖哪些脚本或模板、如何验证结果、什么时候不要使用。

Matdance 对技能做了三层维护：

- 主 agent 在完成任务后，可以主动创建或更新技能。
- 技能整理 subagent 会从已经发生的会话中提取可复用方法。
- 验证与修复 subagent 会检查技能能不能独立使用，并在明确安全的情况下返回 skill-local 修复。

技能也可以导出为 zip，方便迁移和传播。导入侧仍走学习并验证，把外部技能包当作不可信材料本地化，而不是直接覆盖本地技能目录。

v1.1.18-preview 后，`needs_changes` 和 `invalid` 的技能会继续进入待验证队列。这样技能不会因为一次失败报告就停在半残状态里；只要系统有空闲预算，它就可以继续被验证、被修复、再验证。

## 后台任务变成可恢复工作

Matdance 的后台任务不是“开个线程然后希望它别死”。定时任务启动时会先写 active run 和 run 文件；执行中会刷新 heartbeat；模型重试、工具调用、子任务阶段变化和通知投递都会进入诊断链路。

如果 Web UI 关闭、服务器断电、电脑休眠或进程被系统回收，调度器恢复后会补偿错过的触发点。用户创建的定时任务按计划时间补偿；系统内置记忆整理和技能整理按 `agent + taskId` 语义折叠补偿，一次整理覆盖停机期间的待整理变化。技能验证不走启动补偿，它继续在用户空闲时按当前 skill 状态逐个触发。

为了避免长时间停机后一次性把系统打爆，每个任务、每个 agent、每轮 hook 都有限额。这个机制承认中断会发生，然后用文件状态和去重边界把损失压低。

## 工具变成受控动作

Matdance 的工具不是“模型想象自己做过什么”的装饰。浏览器自动化、文件读写、bash、TTS、图像生成、记忆写入、技能编辑、定时任务创建都会留下结果。工具的价值不只是执行动作，更是把动作放进可审计的系统边界里。

这也是为什么 Matdance 把 cookie、隐私访问、Matdance 源码、运行队列和 agent 配置分开处理。系统希望 agent 能做事，但不希望 agent 变成任意修改宿主系统的传话筒。能力和边界必须一起存在，否则长期运行只会把小错误放大成状态污染。

## Web UI 变成控制台

Web UI 是主入口，因为 Matdance 的核心工作流已经超过了普通 CLI 聊天：

- Chat 显示流式回复、工具调用、任务步骤、文件预览、提示音状态和浏览器画面。
- Agent 配置模型、API、窗口、预算和头像。
- Memory 查看、编辑、整理分层记忆，并检索向量索引。
- Skills 管理技能、导出技能、导入外部材料、验证和修复技能。
- Schedule 管理定时任务、运行历史、补偿、重试和修复重试。
- Settings 管理语言、隐私访问、后台事件、提示音和多模态端点。
- Lab 直接测试图像、语音和录音链路，拆穿配置幻觉。

CLI 仍然重要，但它更像启动器、维修口和脚本入口。日常使用应该走 Web UI，因为这个系统需要被看见。
虽然源码内仍有一部分旧 CLI 界面的残留，但不代表开发者推荐使用 CLI，或者 CLI 本身界面就是完整的。
