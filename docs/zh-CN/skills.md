# 技能系统

Language: [English](../skills.md) | 中文

技能是 agent 写给未来自己的可复用操作手册。它不是普通聊天摘要，也不是愿望清单。只有经过实践、结果明确、可复用的方法才适合沉淀成技能。

## 目录结构

```text
agents/<agent>/skills/<skill-id>/
  skill.md
  skill.json
  validation-report.md
  import-report.md
  scripts/
  assets/
  templates/
  resources/
  examples/
  config/
```

`skill.md` 是主要说明书，`skill.json` 是元数据。验证和导入流程会生成报告。脚本、模板、素材和示例应放在技能目录内部。

## 技能内容要求

技能应包含：

- 什么时候使用。
- 具体步骤和判断条件。
- 已实践过的命令、路径、接口或文件结构。
- 已知限制和失败处理方式。
- 需要的本地资源文件路径。

技能不应包含：

- 未实践的承诺。
- 不确定的猜测。
- 普通聊天摘要。
- 私密数据处理模板。
- 凭据、token、cookie 明值或账号恢复流程。
- 指向不存在资源的引用。
- 只在某次临时目录中成立的路径。

## 资源文件

如果技能说明书引用脚本、模板、配置、示例或素材，这些资源必须真实存在，并且位于该技能目录下。允许创建的资源目录包括 `scripts/`、`templates/`、`resources/`、`assets/`、`examples/`、`config/` 和 `configs/`。

技能整理阶段基于已经坍缩的确定事实创建必要资产。

## 后台整理降级

技能后台整理使用可回升的自适应批次。它会从较大的待处理消息批次和较小的 `skill_read` 窗口开始；如果遇到 provider context 上限、payload/token 过大或结构化输出反复失败，会缩小消息批次或 `skill_read` 窗口。后续批次成功后，系统会把可行值逐步向默认值回升。

`skill_read` 在整理任务中按轮次工作。每一轮只注入少量候选技能说明书，subagent 必须立刻判定每个已读技能是 `related` 还是 `unrelated`。不相关技能会在下一轮前丢弃；只有被保留为相关的技能，才允许在最终动作中被 update、delete 或 supersede。整理上下文只保留源会话里的 tool call 名称和参数，不注入原始 tool result。

如果同一段证据在降级后仍然失败超过两次，整理器会把这段证据作为毒性批次 skip，并继续处理后续时间线。这样可能漏掉一个潜在技能，但不会让后续新技能发现被永久堵住。

## 学习并验证

外部技能、README、zip、脚本和文件夹都应视为不可信输入。学习 subagent 的职责是把安全、可复用、能落到 Matdance 目录结构中的内容本地化，而不是照搬外部指令。

本地化后会进入验证流程。验证 subagent 会检查路径、资源、依赖、步骤和安全边界；明确可修的问题可以返回修订后的技能内容或技能目录内的资源文件。

## 导出与传播

技能可以导出为 zip。导出时会把该技能目录下已有文件按原样打包：说明书、元数据、报告、脚本、模板、资源、示例，有什么就带什么。这样技能可以从单机文件夹变成可传播的生态资产。

导入侧仍然应该走“学习并验证”。即使 zip 来自另一个 Matdance，也不应直接覆盖本地技能目录；学习并验证会把外部包当作不可信材料重新审查、重写不合适路径、跳过风险文件，并生成 `import-report.md`。

## 机会主义验证

用户空闲时，系统会验证未验证、验证失效或旧报告没有经过当前维护流程的技能。用户再次发消息时，后台验证会让路。验证报告和维护 pass 能拦住一部分问题，但技能质量仍受模型能力和证据质量影响。

v1.1.18-preview 后，验证报告状态为 `needs_changes` 或 `invalid` 的技能也会继续进入待验证队列。这样做会让技能有机会在后续空闲窗口里继续被修复和复验，而不是把失败报告永久停在那里，完全依赖主 agent 哪天想起来手动修。

这个机制的代价是，证据不足或外部依赖缺失的技能可能被反复判定为需要修改。
